## 2025-05-18 - Layout Thrashing in Marquee Selection
**Learning:** Calling `getBoundingClientRect()` on multiple DOM elements during high-frequency events like `mousemove` causes layout thrashing, as the browser is forced to recalculate styles and layout for every call. In ryOS, this was occurring in `Desktop.tsx` and `FileList.tsx` during marquee selection, where every selectable item was measured on every mouse move.
**Action:** Cache DOM measurements in a `useRef` at the start of the interaction (e.g., on `mousedown`). Since items are unlikely to move or resize *during* a marquee selection, these cached values remain valid until the interaction ends (`mouseup`), at which point the cache should be cleared. This reduces $O(N)$ layout-blocking calls to a single $O(N)$ measurement phase per selection action.
